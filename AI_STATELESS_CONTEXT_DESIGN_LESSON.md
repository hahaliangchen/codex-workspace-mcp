# AI 代理设计复盘：优先借上下文，不急着维护状态

## 背景

在多模态代理适配里，最初的问题是：文本模型不支持图片，但 Codex 请求里可能带图片。为了让 DeepSeek、Mimo 等文本模型也能处理图片，我们做了一个视觉子代理：先分析图片，再把分析结果合并进文本上下文。

早期方案很自然地引入了图片映射：

- 发现图片后生成 `image_key`
- 将 `image_key -> 图片 base64/path` 存入内存或 SQLite
- 后续模型如果要重新看图，就通过 `analyze_image(image_key)` 找回原图

这个方案看起来完整，但实际上引入了复杂状态。

## 旧方案的问题

内存保存会遇到重启丢失：

- 代理重启后 `image_key` 还在历史文本里，但内存映射已经没了
- 模型看到旧 key 后调用工具，工具找不到图片
- 历史中残留的错误信息又会继续污染后续请求

SQLite 保存会遇到生命周期问题：

- 原图什么时候删？
- 是否要清理过期图片？
- base64 体积可能很大
- 保存用户图片有隐私和数据膨胀风险
- 旧 key 会让模型误以为任何历史图片都可以永久访问

更重要的是，这个方案让代理承担了一个本不该承担的职责：长期管理图片资源。

## 思维惯性

这类问题里，AI 很容易顺着已有设计继续补丁式加固：

- 内存丢了，那就加 SQLite
- SQLite 需要清理，那就加 TTL
- key 对不上，那就加恢复逻辑
- 历史污染，那就加更多过滤规则

这些修补都可能局部有效，但会让系统越来越重。

真正应该先问的是：

> 这个状态真的必须由代理维护吗？还是可以借用上游协议、当前请求上下文、模型已有历史能力来完成？

## 新方案

新策略是：图片是当前请求上下文里的资源，不是代理持久状态。

具体规则：

- `raw_codex` provider 直接透传原始请求，让上游模型自己处理多模态
- 非 raw 文本 provider 只处理最新 user 消息里的真实图片
- 视觉子代理分析图片后，把图片块替换成文本形式的 `[图像分析报告]`
- 原始图片 base64/path 不写入文本模型上下文
- 不再输出长期有效的 `[图片 Key]`
- 不再维护 `image_key -> 原图` 的 SQLite/内存持久映射
- `analyze_image` 只尝试分析当前请求上下文里仍可见的原图
- 如果当前上下文里找不到原图，就让用户重新上传

这样，后续文本模型可以直接基于图像分析报告继续对话。只有当用户明确要求“重新看图、看细节、OCR、检查之前没看清的地方”时，才尝试调用视觉工具重新分析当前上下文中的原图。

## 为什么更好

这个方案的关键优势是减少状态承诺。

- 不怕代理重启，因为不依赖长期内存映射
- 不需要 SQLite 保存原图，也没有删除策略负担
- 历史更干净，不会被旧 `image_key` 诱导反复调用工具
- 文本模型只看到分析文本，避免多模态格式不兼容
- raw provider 保留原生多模态能力，能借多少借多少
- 失败路径清晰：找不到原图就请用户重新上传

## 通用原则

遇到代理系统设计问题时，优先按这个顺序思考：

1. 能不能借当前请求上下文？
2. 能不能借上游协议或模型原生能力？
3. 能不能把一次性资源转成文本结果沉淀到历史？
4. 是否真的需要代理维护长期状态？
5. 如果必须维护状态，生命周期、清理、安全边界是否明确？

不要一开始就默认“缺什么状态就补什么状态”。很多时候，更好的设计是重新定义资源边界，让状态自然消失。

## 这个案例的结论

旧方案是在修补状态。

新方案是移除不必要的状态。

这就是区别。
